# Section 10 — Risk Register

| Risk ID | Risk Description | Category | Probability | Impact | Mitigation Strategy | Owner |
|---------|-----------------|----------|-------------|--------|---------------------|-------|
| R-01 | **Luma AI API unavailability or deprecation** — The 3D reconstruction API may not have a publicly available endpoint, may change pricing, or may be deprecated during the project timeline. | Technical | HIGH | HIGH | Primary: Implement Tripo AI as an alternative cloud-based 3D API. Fallback: Implement COLMAP as a self-hosted alternative (runs on server, no external API dependency). Architecture uses a service abstraction layer — the reconstruction service interface remains the same regardless of which backend provider processes the job. | Tech Lead |
| R-02 | **OpenAI API cost overruns** — Unexpected high volume of AI search queries could generate API costs exceeding the $60/month budget. GPT-4o-mini costs ~$0.15/1M input tokens, but system prompts with the 200+ term glossary add ~500 tokens per request. | Financial | MEDIUM | MEDIUM | Implement aggressive caching: cache parsed query results in Redis (TTL 1 hour) keyed by query hash. Identical or near-identical queries hit cache instead of OpenAI. Set monthly API budget alerts in OpenAI dashboard. Degrade gracefully: if API budget is exhausted, fall back to regex-based keyword extraction for basic queries (rooms, price, district). | Tech Lead |
| R-03 | **OSM data quality issues in Tashkent** — Key POIs (bus stops, parks, schools) may be unmapped or incorrectly tagged in OSM for certain neighborhoods, leading to inaccurate comfort scores. | Data | HIGH | MEDIUM | Display confidence indicators (LOW/MEDIUM/HIGH) on all comfort scores based on the number of data points within the scoring radius. Low-confidence scores show a prominent warning. Supplement OSM with Google Places data for categories where coverage is better (shops, restaurants, schools). Implement a "data quality" audit script that compares OSM POI counts against expected baselines per district. | Tech Lead |
| R-04 | **No partner platform commits to pilot integration** — B2B sales cycles are long, and platform operators may be unwilling to integrate an unproven technology from an unknown provider within the 100-day timeline. | Business | HIGH | HIGH | Build the MVP as a standalone showcase accessible via a public URL. Create a video demo (screen recording with narration) demonstrating the integration flow. Prepare a one-page commercial proposal summarizing expected engagement improvements. Target smaller platforms (Birbir.uz) first where decision-making is faster. Consider approaching real estate agencies (not just platforms) as alternative early adopters. | Project Lead |
| R-05 | **3D model file sizes exceed 10 MB target** — Complex properties with many rooms or detailed furniture may produce GLB files larger than 10 MB, causing slow load times on mobile/4G connections. | Technical | MEDIUM | MEDIUM | Implement mesh decimation as a post-processing step (reduce polygon count by 50–70% while preserving visual quality). Use Draco compression for GLB files (can reduce file size by 80%+). Set a hard upload validation: reject or reprocess models exceeding 10 MB. Provide loading progress indicators to manage user expectations during slow downloads. | Tech Lead |
| R-06 | **Single-instance server failure causes complete outage** — AWS Lightsail instance hardware failure, OOM kill, or misconfiguration results in total service unavailability with no automatic failover. | Infrastructure | LOW | HIGH | Configure daily automated snapshots on Lightsail (retention: 7 days). Document a disaster recovery runbook: restore from snapshot → reconfigure DNS → verify services. Estimated recovery time: 30–60 minutes. Monitor instance health with Lightsail built-in monitoring + UptimeRobot (free external monitoring). Post-MVP: migrate to multi-instance deployment with load balancer. | Tech Lead |
| R-07 | **Google Places API free tier exhaustion** — The 200 requests/day free tier may be insufficient to seed POI data for all 30 demo properties if many requests are needed per property. | Technical | MEDIUM | LOW | Batch POI fetching: fetch all POIs within the entire Tashkent bounding box in one query rather than per-property queries. Cache all Google Places results in the `pois` table — subsequent comfort score computations use the local database, not live API calls. Minimize live Google Places calls to initial data seeding + weekly refresh only. Use OSM as the primary data source for categories available on both platforms. | Tech Lead |
| R-08 | **Uzbek AI query parsing fails below 50% accuracy** — The RAG glossary approach may be insufficient to handle complex or colloquial Uzbek real estate queries, resulting in a poor user experience for Uzbek-speaking users. | Technical | MEDIUM | MEDIUM | Build the RAG glossary iteratively: start with 200+ terms, expand based on testing results. Include 10+ Uzbek query examples as few-shot demonstrations in the system prompt. If accuracy remains below acceptable thresholds, add a "Did you mean...?" confirmation step that shows the parsed filters before executing the search, allowing users to correct misinterpretations. Promote Russian/English search as the primary mode in MVP UI. | Tech Lead |
| R-09 | **Data privacy regulatory action** — Uzbek data protection authorities could interpret PropVision's collection of geospatial property data or comfort analytics as regulated activity, triggering compliance requirements. | Legal | LOW | MEDIUM | Proactive compliance: ensure no personally identifiable information (PII) is collected, stored, or processed. API logs record partner API key (not user data). Comfort analytics use only publicly available data (OSM, Google Places). Document compliance posture in a data protection impact assessment (DPIA). Engage local legal counsel before any feature involving user-identifiable data is launched. | Project Lead |
| R-10 | **Frontend performance degradation on mobile devices** — The combination of Mapbox GL JS (WebGL map), Three.js (WebGL 3D viewer), and Recharts (SVG radar chart) may exceed GPU/CPU capabilities of mid-range Android devices common in Uzbekistan. | Technical | MEDIUM | MEDIUM | Profile performance on target devices (Samsung Galaxy A-series, Xiaomi Redmi series). Implement lazy loading: 3D viewer initializes only when the user clicks "View 3D Model," not on page load. Use Mapbox's `maxPitch: 45` and `antialias: false` on mobile to reduce GPU load. Compress textures to WebP/AVIF format. Implement progressive enhancement: basic map view for low-end devices, full 3D features for capable devices. | Tech Lead |
| R-11 | **Scope creep during 100-day timeline** — The comprehensive scope (3D + AI + comfort + widget + dashboard) risks incomplete delivery if any component takes longer than estimated. | Project | HIGH | MEDIUM | Strictly prioritize MoSCoW: Must-have (map viewer, AI search, comfort scores), Should-have (3D reconstruction, widget), Could-have (analytics dashboard, partner dashboard mockup). If schedule slips, cut Could-have items first. Track velocity in Jira — if actual velocity < planned velocity by Sprint 4, re-evaluate scope. | Project Lead |
| R-12 | **Cross-browser compatibility issues with WebGL** — Three.js and Mapbox GL JS both depend on WebGL, which has inconsistent support across browser versions, especially on older Safari and mobile browsers. | Technical | MEDIUM | LOW | Test on target browsers early (Sprint 5): Chrome 120+, Firefox 120+, Safari 17+, Edge 120+. Implement feature detection: if `WebGLRenderingContext` is not supported, hide 3D viewer and show photo gallery. Mapbox GL JS includes built-in fallback detection. Document minimum browser requirements in the user-facing UI. | Tech Lead |
