# Section 3 — Scope and Deliverables

## 3.1 Core Functionality (Including AI Search)

| Category | In-Scope (MVP) — Specific Definition | Out-of-Scope | Definition of Done |
|----------|---------------------------------------|--------------|-------------------|
| **Interactive 3D Map Viewer** | A full-screen web map rendered with **Mapbox GL JS v3** displaying **30 pre-loaded demo properties** in Tashkent as interactive markers. The map supports: **zoom** (levels 10–18), **pan** (drag to navigate), **tilt** (3D perspective), **marker clustering** (auto-cluster when >5 markers overlap at current zoom), and **district-level filtering** (click district boundary to filter). Clicking a marker opens a **PropertyPanel** side panel showing property photos, price (USD), room count, area (m²), address, comfort scores (radar chart), and an embedded 3D model viewer (if available). The map includes **building extrusions** (Mapbox 3D buildings layer) for spatial context. | Custom map style design; satellite imagery toggle; street-level view; real-time property position tracking; offline map support; non-Tashkent regions. | Map loads within 2 seconds on 10 Mbps. All 30 markers render correctly. Clicking any marker opens the property detail panel with correct data. District filtering reduces visible markers to the selected district. Zoom/pan/tilt interactions are smooth (60 FPS on modern hardware). |
| **AI Natural Language Property Search** | A search bar prominently placed above the map where users type **free-form natural language queries** in English, Russian, or Uzbek. The system processes each query through the following pipeline: **(1)** Client sends the raw query string to `POST /api/v1/search`. **(2)** The backend calls **OpenAI GPT-4o-mini** with a structured output prompt that includes a RAG context glossary of 200+ Uzbek real estate terms. **(3)** The AI returns structured filters: `rooms` (int), `min_price`/`max_price` (float, USD), `district` (string), `proximity_to` (enum: metro, park, school, shop, hospital), `max_distance_m` (int). **(4)** These filters execute as **PostGIS spatial queries** using `ST_DWithin` for proximity and indexed B-tree queries for price/rooms. **(5)** Results are returned ranked by relevance (proximity match → price match → recency) and displayed as: highlighted map markers with fly-to animation + scrollable result list on the side panel. **Supported query types:** room count, price range (USD), district name, proximity to POI types, area range (m²). | Conversational memory (multi-turn dialogue); voice input; image-based search ("find apartments that look like this"); recommendation engine; saved searches; search history; user preference learning. | 50 test queries (20 English, 20 Russian, 10 mixed Uzbek) achieve ≥85% parse accuracy. Search results return within 2 seconds. Fly-to animation plays correctly on result selection. Empty results display a "No properties found" message with suggested filter relaxation. |
| **Comfort Analytics Dashboard** | For each of the 30 demo properties, the system computes and displays **5 livability scores** (0–100 each): **Transport Score** (bus stops, metro stations, taxi stands within 1 km, from OSM Overpass API), **Shopping Score** (supermarkets, convenience stores, markets within 800 m, from Google Places API + OSM), **Education Score** (schools, kindergartens, universities within 1.5 km, from Google Places API + OSM), **Green Space Score** (parks, gardens, recreational zones within 1 km, from OSM land-use data), **Safety Score** (approximate: street lighting density, police stations, hospitals within 1 km, from OSM). An **Overall Comfort Index** is computed as a weighted average (transport 25%, shopping 20%, education 20%, green space 15%, safety 20%). Scores are visualized as: **(a)** a **Recharts radar chart** on the property detail panel, **(b)** **colored map overlays** (heatmap-style, green→yellow→red gradient) showing comfort score distribution across the map. Each score displays a **confidence indicator** (HIGH/MEDIUM/LOW) based on the number of data points available. Scores are updated on a **nightly batch schedule** (cron job), not in real-time. | Real-time score computation; user-customizable weights (MVP uses fixed weights); noise level data; air quality data; crime statistics; historical score trends; neighborhood comparison tool. | All 30 properties have non-null scores for all 5 dimensions. Radar chart renders correctly with labeled axes. Heatmap overlay toggles on/off. Confidence warning displays for properties with LOW confidence. Scores are recomputed successfully by the nightly cron job. |
| **3D Property Preview (Prototype)** | For **5 of the 30 demo properties**, generate a 3D reconstruction from uploaded photos using the **Luma AI API** (primary) or **COLMAP** (self-hosted fallback). The pipeline: **(1)** accepts 8–15 JPEG/PNG images per property via `POST /api/v1/3d/upload` (max 10 MB per image), **(2)** sends to the reconstruction API, **(3)** polls for completion (typical: 5–20 minutes), **(4)** downloads the resulting **GLB file** (target: ≤ 10 MB), **(5)** stores in S3-compatible object storage. The GLB model is displayed in an embedded **Three.js viewer** (using `@react-three/fiber`) with: **OrbitControls** (rotate, zoom, pan), **auto-rotation toggle**, **fullscreen mode**, and **loading progress indicator**. For the remaining **25 properties** without 3D models: display a **responsive photo gallery** (CSS carousel with swipe support) as a fallback. | Professional-grade 3D scans (Matterport quality); floor plan extraction from 3D model; measurement tools; VR/AR viewing; video walkthroughs; real-time collaborative model viewing; model editing. | 5 demo properties have viewable 3D models in the browser. GLB files are ≤ 10 MB each. Three.js viewer loads model within 5 seconds. OrbitControls function correctly (rotate, zoom, pan). Fullscreen mode works. Fallback photo gallery displays for properties without 3D models. Upload endpoint validates file count (8–15) and file types (JPEG/PNG). |

## 3.2 Integration & APIs

| Category | In-Scope (MVP) — Specific Definition | Out-of-Scope | Definition of Done |
|----------|---------------------------------------|--------------|-------------------|
| **REST API** | A fully documented **OpenAPI 3.0** specification with the following endpoints: `POST /api/v1/properties` (ingest property from partner), `GET /api/v1/properties` (list/filter with bbox, price, rooms, district), `GET /api/v1/properties/{id}` (single property with comfort scores), `POST /api/v1/search` (AI NLP search), `GET /api/v1/comfort/{property_id}` (comfort score breakdown), `POST /api/v1/3d/upload` (photo upload for 3D), `GET /api/v1/3d/{property_id}/status` (reconstruction job status), `GET /api/v1/analytics/dashboard` (engagement metrics). **Authentication**: API key passed via `X-API-Key` header. Each partner gets a unique API key. Key validation is done by hashing and comparing against the `partners` table. **Error handling**: All endpoints return standardized JSON error responses with `status_code`, `detail`, and `error_code` fields. **Rate limiting**: 100 requests/minute per API key, enforced via Redis sliding window counter. | OAuth 2.0 authentication; JWT tokens; API versioning beyond v1; GraphQL; WebSocket endpoints; webhook delivery; API gateway (Kong, AWS API Gateway); API key self-service portal. | Swagger UI accessible at `/api/v1/docs`. All 8 endpoints respond correctly to valid requests. Invalid API key returns 401. Missing required fields return 422 with descriptive error. Rate limit exceeded returns 429. OpenAPI spec validates against `swagger-parser`. |
| **Embeddable Widget** | A standalone **JavaScript file** (`widget.js`, ≤ 50 KB minified) that partner platforms include via `<script src="https://api.propvision.ai/widget.js"></script>`. The script reads configuration from a `<div id="propvision-widget" data-api-key="..." data-property-id="..." data-theme="light|dark"></div>` element on the partner's page. It creates an **iframe** pointing to the PropVision frontend with the property detail view (comfort panel + 3D viewer). The widget: **(a)** handles responsive sizing (fills container width, min-height 400px, max-height 800px), **(b)** supports theme configuration (light/dark mode via data attribute), **(c)** uses cross-origin messaging (`postMessage`) for bidirectional communication (e.g., partner page can update `property_id`, widget can notify partner of user interactions). | Multi-widget per page; widget SDK with programmatic API (mountWidget, destroyWidget, etc.); custom CSS injection; server-side rendering of widget; widget analytics tracking; A/B testing framework. | Widget renders correctly in Chrome, Firefox, Safari, and Edge. Widget loads in ≤ 3 seconds. Light and dark themes display correctly. Responsive sizing works from 320px to 1920px container width. Cross-origin messaging works for property ID updates. |
| **Data Ingestion** | Partners push property data via `POST /api/v1/properties` with a **flexible JSON body**. The API accepts a superset of possible fields and maps them to the internal schema using a **per-partner field mapping configuration** stored as JSONB in the `partners` table. Example: Partner A sends `{"apartment_title": "...", "cost_usd": 50000}` and their field mapping config maps `apartment_title → title`, `cost_usd → price`. Supported internal fields: `title`, `description`, `price`, `rooms`, `area_sqm`, `district`, `address`, `latitude`, `longitude`, `photos` (array of URLs). | Bulk import via CSV/Excel; real-time data sync (webhooks from partner to PropVision); duplicate detection; automated photo scraping from partner listings; data transformation rules beyond field renaming. | Property data from a partner with custom field names is correctly ingested and stored with internal field names. Missing required fields (title, price, latitude, longitude) return 422 error. Duplicate property detection by (partner_id, external_id) prevents duplicates. |

## 3.4 3D Visualization & Processing

| Category | In-Scope (MVP) — Specific Definition | Out-of-Scope | Definition of Done |
|----------|---------------------------------------|--------------|-------------------|
| **Photo-to-3D Pipeline** | An asynchronous processing pipeline: **(1)** Upload endpoint accepts 8–15 JPEG/PNG images (max 10 MB each) via multipart form upload. **(2)** Images are stored in S3-compatible object storage (AWS Lightsail). **(3)** A reconstruction job is submitted to the **Luma AI API** (primary) or processed locally via **COLMAP** (fallback). **(4)** The system polls for job completion (5–20 min typical). **(5)** Upon completion, the resulting GLB file is downloaded and stored in S3. **(6)** The property record is updated with the `model_3d_url`. **(7)** Job status is tracked: `pending → processing → completed → failed`. | Real-time processing; video-based reconstruction; LiDAR point cloud ingestion; mesh editing; texture enhancement; multi-property batch processing; GPU-accelerated local processing. | 5 demo properties processed successfully. Job status transitions are tracked correctly. Failed jobs show descriptive error messages. GLB files are accessible via S3 URL. Upload validation rejects files outside 8–15 count range and non-image files. |
| **3D Viewer** | A **Three.js-based viewer** (using `@react-three/fiber` and `@react-three/drei`) embedded in the property detail panel. Features: **OrbitControls** (mouse/touch: rotate, scroll: zoom, right-click: pan), **auto-rotation toggle** (button to start/stop automatic y-axis rotation at 1 RPM), **fullscreen mode** (enter/exit via button), **loading progress bar** (shows percentage while GLB is downloading). The viewer has a neutral gray background with soft ambient lighting. **Fallback**: if no 3D model is available, display a **responsive photo carousel** with left/right navigation arrows and dot indicators. | Annotations/measurements on 3D model; floor plan view; cross-section cutting planes; material/texture replacement; AR mode; VR headset support; collaborative viewing; screenshot/share of 3D view. | 3D viewer loads and renders GLB models correctly. OrbitControls respond to mouse and touch inputs. Auto-rotation toggles on/off. Fullscreen works in Chrome, Firefox, Safari, Edge. Loading progress bar displays during model download. Photo carousel fallback displays when no 3D model exists. |
